CRITICAL GRAMMAR DESIGN AND GENERATION CONSTRAINTS:

You MUST follow these constraints to create grammars that LLMs can easily generate models from:

AVOID (Metrics that significantly hinder model generation):
- Comment lines in grammar: CRITICAL - Comments are the strongest predictor of generation failure for simple models (r = -0.268). COMPLETELY AVOID comments, documentation, or explanatory text within the .tx grammar file.
- High number of grammar rules (num_rules): For optimal simple model generation, keep under 60 rules (correlation r = -0.189). Each additional rule beyond this threshold reduces model generation success.
- High number of non-terminal symbols (num_non_terminals): Limit to under 50 non-terminals (correlation r = -0.179). Non-terminals represent abstract grammar concepts that increase cognitive load for LLMs.
- Excessive choice operators (number_of_choices): Minimize use of '|' operators. Each choice point creates branching complexity that confuses model generation.
- High Halstead complexity metrics:
  * Halstead effort (computational complexity): Keep under 500,000
  * Halstead vocabulary (unique operators + operands): Minimize distinct grammar symbols
  * Halstead volume (program length × log vocabulary): Reduce overall grammar size and symbol diversity

FAVOR (Metrics that improve model generation success):
- Higher terminal-to-nonterminal ratio: PRIORITIZE ratios above 3.5. This is the strongest positive predictor for simple model success (r = +0.136). Use concrete string literals, keywords, and tokens extensively rather than abstract rule names.
- Higher graph density: Create more interconnected grammar structures where rules reference each other, rather than isolated rule clusters (r = +0.160).

METRIC DEFINITIONS:
- num_rules: Total count of grammar production rules (left-hand side rule definitions)
- num_non_terminals: Count of abstract rule names that appear on left-hand sides
- number_of_choices: Count of '|' choice operators throughout the grammar
- terminal_to_nonterminal_ratio: Ratio of concrete tokens to abstract rule names
- graph_density: Measure of how interconnected grammar rules are (0-1 scale)
- halstead_effort: Computational complexity metric combining program length, vocabulary, and difficulty
- comment_lines: Number of lines containing grammar comments

IMPLEMENTATION STRATEGY:
- Design fewer, more comprehensive rules rather than many small ones (stay under 60 total rules)
- Use concrete string literals and keywords instead of excessive rule abstractions
- Minimize branching with '|' operators - prefer sequential rule structures
- Create rules that reference other rules to increase interconnectedness
- Produce a completely uncommented grammar file - no inline documentation whatsoever

/*
    Specification of textX grammar language in textX.
*/

TextxModel:
    imports_or_references*=ImportOrReferenceStm
    rules*=TextxRule
;

ImportOrReferenceStm:
    ImportStm | ReferenceStm
;

ImportStm:
    'import' grammar_to_import=GrammarToImport
;

ReferenceStm:
    'reference' language_name=LanguageName ('as' language_alias=ID)?
;

GrammarToImport:
    /(\w|\.)+/
;

LanguageName:
    /(\w|-)+/
;

// Rules
TextxRule:
    name=ID ('[' params+=RuleParam[','] ']')? ':'
        body=TextxRuleBody
    ';'
;

RuleParam:
    name=ID ('=' value=STRING)?
;

TextxRuleBody:
    Choice
;

Choice:
    sequences+=Sequence['|']
;

Sequence:
    repeatable_exprs+=RepeatableExpr
;

RepeatableExpr:
    expr=Expression operator=RepeatOperator? ('-')?
;

Expression:
    assignment=Assignment
    | (predicate=SyntacticPredicate?
       (simple_match=SimpleMatch
        | rule_ref=RuleRef
        | bracketed_choice=BracketedChoice ))
;


BracketedChoice:
    '(' choice=Choice ')'
;

RepeatOperator:
    op_sign=OperatorSign ('[' modifiers=RepeatModifiers ']')?
;

OperatorSign:
	"*" | "?" | "+" | "#"
;

RepeatModifiers:
      (SimpleMatch+ | 'eolterm')
;


SyntacticPredicate:
    "!" | "&"
;

SimpleMatch:
    StrMatch | ReMatch
;

StrMatch:
    match=STRING
;

ReMatch:
    '/' match=/((\\/)|[^\/])*/ '/'
;


// Assignment
Assignment:
    attribute=ID operator=AssignmentOp rhs=AssignmentRhs
;

AssignmentOp:
    "=" | "*=" | "+=" | "?="
;

AssignmentRhs:
    (simple=SimpleMatch | reference=Reference)  ('[' modifiers=RepeatModifiers ']')?
;

// References
Reference:
    rule=RuleRef | obj=ObjRef
;

RuleRef:
    builtin=BuiltinRuleRef | plain=ID
;

BuiltinRuleRef:
    'ID' | 'BOOL' | 'INT' | 'FLOAT' | 'STRING' | 'NUMBER' | 'BASETYPE'
;

ObjRef: // TODO: Remove '|' separator in 4.0
    '[' name=ClassName ((':' | '|') obj_ref_rule=ID ('|' rrel=RRELExpression))? ']'
;

ClassName:
    QualifiedIdent
;


// RREL
RRELExpression: multi?="+m:" sequence=RRELSequence;
RRELSequence: paths+=RRELPath[','];
RRELPath: ((up='^' | dots=RRELDots)? parts+=RRELPathPart['.']) | (up='^' | dots=RRELDots);
RRELPathPart: RRELZeroOrMore | RRELPathElement;
RRELZeroOrMore: element=RRELPathElement '*';
RRELPathElement: RRELParent | RRELBrackets | RRELNavigation;
RRELDots: /\.+/;
RRELBrackets: '(' sequence=RRELSequence ')';
RRELNavigation: noconsume?='~' attr=ID;
RRELParent: 'parent' '(' type=ID ')';


QualifiedIdent:
    /\w+(\.\w+)?/
;

// Comments
Comment:
    CommentLine | CommentBlock
;

CommentLine:
    /\/\/.*?$/
;

CommentBlock:
    /\/\*(.|\n)*?\*\//
;

These are some common textx syntax errors that you should avoid:

1) Cannot declare multiple delimiters in a single rule using the '|' operator.
You can use only one delimiter as below
numbers*=INT[',']
You cannot use it like this:
numbers*=INT[',' | '+']

2) Cannot reference undefined classes/rules in your grammar.
All rule references must be defined somewhere in the grammar.
Example of INCORRECT usage:
MyRule: items*=UndefinedRule;
This will cause: Semantic error: Unknown class/rule "UndefinedRule"
Example of CORRECT usage:
MyRule: items*=DefinedRule;
DefinedRule: name=ID;
CRITICAL: Ensure all rule references in your grammar are properly defined.
